📘 2. Facebook — Posting & Engagement

You already have facebook_bot/ with messenger conversations, lead scoring, and comment handling. ~80% of infrastructure exists.

Existing Code Advantage Your codebase already includes fb_leads, fb_messages, fb_comment_replies, fb_sequences, and fb_agent_alerts tables — plus a working Messenger bot.

Implementation — Graph API (No Browser Automation Needed)

# Posting POST /{page-id}/feed → Text posts POST /{page-id}/photos → Image posts POST /{page-id}/videos → Video posts (Reels via /video_reels) # Engagement POST /{post-id}/likes → Like a post POST /{post-id}/comments → Comment on a post GET /{page-id}/feed → Read feed for engagement targets

Key Considerations

⚠ Requires Facebook Page (not personal profile) — personal profile automation violates ToS and has no API.
⚠ Need Page Access Token with pages_manage_posts, pages_read_engagement permissions.
⚠ App Review required by Meta for publishing permissions (1–4 weeks approval).
✓ Rate limit: 200 calls/user/hour for Graph API.
⚠ Facebook's algorithm penalizes automated/spammy behavior — need human-like timing.

What's Missing From Your Existing Facebook Code

✗ Post creation (current code is lead-gen focused, not content posting)
✗ Feed engagement (like/comment automation)
✗ Analytics/stats tracking
✗ Integration with the WhatsApp command interface

📸 3. Instagram — Posting & Engagement

Instagram Graph API via Meta Business Suite — same developer account as Facebook.

Implementation — Instagram Graph API

# Posting (two-step process) POST /{ig-user-id}/media → Create media container (image/video/carousel) POST /{ig-user-id}/media_publish → Publish the container # Engagement POST /{media-id}/comments → Comment on posts GET /{ig-user-id}/media → Get user's posts # NOTE: No API for liking posts — only comment management

Key Considerations

⚠ Instagram Business/Creator account required (linked to Facebook Page).
✗ No like API — Instagram Graph API does NOT support liking other users' posts. Browser automation needed (risky — Instagram bans aggressively).
✗ No general DM API — only reply to user-initiated DMs via Messenger API.
⚠ Publishing limits: 25 API posts per 24-hour period per account.
✓ Carousel support: Up to 10 images/videos per carousel.
✓ Reels: Supported via /media endpoint with media_type=REELS.
✓ Hashtag research: Available via /ig_hashtag_search endpoint.
⚠ Media hosting: Images must be at a public URL before publishing (upload to S3/CDN first).
⚠ Caption limit: 2,200 characters. Rate limit: 200 calls/user/hour.

📕 4. Rednote (Xiaohongshu / 小红书)

The hardest platform. No public API — 100% browser automation required.

High Risk Platform Rednote has no official API. All automation requires browser automation or reverse-engineering private APIs. Aggressive anti-bot detection and Chinese regulatory considerations add significant risk.

Implementation — Browser Automation Only

# Same approach as LinkedIn Selenium/Playwright → Login → Navigate → Post/Engage → Anti-detection (fingerprinting, captcha solving)

Key Considerations

✗ No official API — 100% browser automation or reverse-engineering private APIs.
✗ Aggressive anti-bot detection: Advanced fingerprinting, behavioral analysis, and CAPTCHA challenges.
⚠ Chinese phone number typically required for account creation.
⚠ Content format: Image-first platform. Text posts need cover images.
✓ Language: Primarily Chinese — Claude handles Chinese content generation well.
✗ Legal risk: Violating ToS in China's regulatory environment carries more risk.
✗ Login flow: Phone/SMS verification, WeChat login — all harder to automate.
⚠ Rate limits: Unknown (no documentation), accounts get banned quickly with automation.
⚠ Captcha solving: Needs a service like 2captcha or hCaptcha solver (~$2–3/1,000 solves).

🗓 Implementation Priority & Timeline

Recommended build order — lowest effort and highest impact first.

Phase 1: Facebook 1.5 weeks Easy Graph API

Lowest effort — you have existing code in facebook_bot/. Graph API is stable and well-documented. Add post creation and feed engagement to complement existing Messenger/lead-gen features.

Phase 2: Instagram 3 weeks Medium Graph API

Same Meta developer account as Facebook — shares authentication infrastructure. Two-step publishing (create container → publish). Note: no like API, would need browser automation for liking.

Phase 3: WhatsApp 2–3 weeks Medium Cloud API

Can run in parallel with platform work. Same Meta Business account. Replaces Telegram command interface. Requires rebuilding conversation state machine and adapting to 3-button limit.

Phase 4: Rednote 5–6 weeks Hard Browser Automation

Highest risk and effort. No public API — requires browser automation with anti-detection measures. Consider dropping if margins don't justify the maintenance burden and ban risk.

🏗 Architectural Changes Required

What needs to be redesigned vs what works as-is.

Works Well (Keep As-Is)

✓ Multi-user task queue (Celery) — scales horizontally
✓ Browser pool management — resource efficient
✓ Database schema — flexible, ready for multi-platform
✓ Payment system — generic Stripe implementation
✓ AI service layer — platform-agnostic Claude integration
✓ Safety limits — prevent account bans

Needs Redesign

✗ Telegram command handlers → abstract command interface
✗ LinkedIn browser automation → API calls where possible
✗ Telegram notifications → platform-specific message templates
✗ User ID mapping → unified user model with platform flag
✗ Conversation state machine → generic state handler
✗ Platform-specific rate limits per social network
Key Schema Change The current users table uses telegram_id as the primary user identifier. For multi-platform, add a platform column (e.g., 'telegram', 'whatsapp') and migrate to a unified user_id + platform composite key. Similarly, add a target_platform column to track which social media each automation targets (LinkedIn, Facebook, Instagram, Rednote).
